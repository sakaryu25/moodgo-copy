# MoodGo 検索フロー詳細（2026-06-14 時点・本番反映済み）

`POST /api/recommend`（[app/api/recommend/route.ts](../app/api/recommend/route.ts)）の処理を上から順に説明する。
中心は **Supabase の自前DB(約28,000件)を主軸 → OpenAIで良い順に判別 → 足りない分だけGoogle/Yahoo → 最終15件だけ写真補完** という構成。

---

## 0. 入口と計測ラッパー

- `POST` 関数（計測ラッパー）が `apiCounterStore`（AsyncLocalStorage）で **Google API呼び出し回数**を数えながら本体 `handleRecommend` を実行。
- 応答に `_apiCount`（searchText / photo / geocode 等）を埋め込み、`search_metrics` テーブルに1検索ごと記録（Google依存度KPI用）。
- 気分/エリアは `request.clone()` で先読み（本処理に影響なし）。

## 1. 入力 → answers

クイズ回答が `answers` に集約：
`mood`（気分）/ `dynamicQs`（深掘り：カテゴリ・詳細）/ `companion`（同行者）/ `distanceFeeling`・`radiusKm`（距離）/ `originLat/Lng` or 手動エリア / `freeWord`（自由ワード）/ `budget` ほか。

## 2. タグ抽出

- **mustTags**：気分タグ（例 `お腹すいた`→`#お腹すいた`）＋深掘りタグ。`MOOD_SHORT_KEY_TO_TAG` / `DRILL_ANSWER_TO_MUST` で変換。
- **niceTags**：付加条件（あると加点）。
- 端末プロファイル（お気に入り・高評価のタグTop5）も `userPreferenceHints` として niceに合流（端末ローカル計算・プライバシー安全）。
- 深掘り指定時は気分タグの fallback を禁止（一般公園・神社の混入を防ぐ）。

## 3. 距離レンジ

- `radiusKm`（上限）＝ `DISTANCE_RADIUS_KM[距離感]`（すぐそこ8km 〜 どこでも160km）。
- `minRadiusKm`（**遠出バイアス**）＝ `DISTANCE_MIN_KM[距離感]`。近すぎる場所を後回しにする。
  - **お腹すいた・心霊は min=0**（最寄り優先）。
- spatialSearch には実 `minRadiusKm` を渡す（0ハードコードは「県またぎでも近場が出る」バグの原因なので禁止）。

## 4. Supabase 本検索（自前DB＝ほぼ無料）

- PostGIS RPC `find_nearby_places(req_tags, lat, lng, radius_m, …)`。タグはAND意味だが、`spatial-search.ts` が複数タグをOR検索でラップ。
- **スコアリング**（`scored`）：
  ```
  _niceScore = (niceタグ一致数)
             + wilsonLower(rating, reviewCount) * 2     // Wilson下限(95%)
             + (手動登録 manual/admin/user なら +5)       // 埋もれ防止で大優先
             + Math.random()*0.3                         // 毎回同じ並びにしない微小乱数
  ```
  - **Wilson下限**：少件数の★5/2件が、多件数の★4.3/800件に勝てないよう信頼区間で補正（`wilsonLower`）。
- **`sbTakeCount = 16`**（通常）/ 15（充足時・心霊）。→ 上位16件をプール化。
- 写真補完対象 `noPhotoNames` は **上位8件のみ**（searchText爆発を防ぐコスト上限）。

## 5. OpenAIによる判別（体験系のみ）

- 16件のSupabase候補を **gpt-4o-mini** が「利用者にとって良い順」に並べ替え（`{"order":[…]}` をJSONで返す）。
  - 条件：`openai && !isFoodMood && !isProprietaryOnly && scored.length > 8`
  - temp 0.2 / max_tokens 240 / **timeout 7s** / 失敗時は元のスコア順を維持。
- 判別結果で `scoredRanked` を作り、これを表示用 `supabaseRecs` の元にする（＝OpenAIが選んだ上位を表示）。

## 6. 充足判定（コスト最適化の要）

- `sbQualified.length >= 15` → **skipAllSupplements**（Google/Yahoo両方呼ばない＝最安）。
- `>= 10` → **skipYahooOnly**（Yahooだけスキップ）。
- 心霊（`isProprietaryOnly`）は常に両方スキップ。

## 7. Google / Yahoo 補足 ＋ 並列処理

`Promise.all` で同時実行：
1. **Google Places 補足**（`fetchGooglePlacesSupplement`, 最大15取得）— 充足時は呼ばない。
2. **Yahooローカルサーチ補足**（`fetchYahooSupplement`, **最大10取得**）— Yahooは写真を返さないので各結果を **Google Places Text Searchで写真補完**（←searchTextのコスト源。表示5件なので10に抑制）。
   - Yahoo仕様：keyword と gc(業種コード) は別々に実行してマージ／複数語は1語ずつ／start=1固定。
3. OpenAI理由生成（自由ワード・絞り込み時のみ）。
4. Supabase写真補完（photo_url空の上位8件をGoogleで補完）。
5. 最寄り駅補完（HeartRails＝無料 → 失敗時Google）。
6. **OpenAI判別**（手順5）。

## 8. 各ソースの品質フィルタ（finalizeSource）

全ソースに同じ後処理パイプライン：
```
applyMallFilter → foodSanitize → seenFilter → qualitySanitize
 → genreFidelityFilter → nonFoodSanitize → sortOrShuffle → ratingSanitize
```
- **qualitySanitize**：B2B(株式会社/工場)・写真なし低評価を除去（※心霊は写真ゲートをスキップ）。
- **genreFidelityFilter**：否定語/肯定語で異ジャンル混入を防ぐ（※心霊はタグ識別なのでスキップ）。
- **nonFoodSanitize**：飲食不要の気分では飲食店を除去（`isFoodAllowedContext`）。
- 飲食×深掘りなしは同一粗ジャンル最大2件（全部ラーメン防止）。

## 9. マージ配分（現行）

```
Supabase 8件 / Google 2件 / Yahoo 5件 を重複排除しながら取得（pickUnique）
不足分の補填順: Supabase余り → Yahoo余り → Google余り （Googleは最後＝最小化）
→ 合計15件
```
- Yahooが0件なら不足分はGoogle余りで埋める（=「足りなければどこかで補う」）。

## 10. 学習による昇格・降格（ratingSanitize / ratingJudge）

- 👎が過半数(3件以上) → **除外** ／ 👎優勢 → 末尾降格 ／ 👍Wilson＋エンゲージメント → 先頭昇格。
- エンゲージメント（`/api/engagement`）：**map_click / detail_view / favorite / visited / share**。貯まるほど良い場所が上位に。
- 学習テーブル：`mood_place_ratings`（👍/👎集計・10分キャッシュ）/ `spot_engagement`（暗黙行動）。

## 11. 最終15件のエンリッチ（写真・営業時間）

- 写真10枚＋営業時間を **最終15件だけ** Googleで補完。
- **ltCache**（`api_cache` テーブル, `enr:<店名>` キー, TTL30日）から先に充当 → 2回目以降はGoogleを呼ばない。
- 取得した写真は **`places.photo_url` に恒久保存**（`schedulePhotoWriteBack` を `after()` で確実実行）→ 次回以降その場所はGoogle不要。

## 12. LLMリランカー（最終並べ替え・体験系のみ）

- 最終15件を gpt-4o-mini が「ぴったり来る順」に再ソート（`!isFoodMood && !isProprietaryOnly && length>=5`）。失敗時は元順。

## 13. 出力

- `{ recommendations(15件), source:"supabase", warning, _apiCount }` を返却。
- `search_metrics` に Google呼び出し数等を記録 → `/api/admin/search-metrics` で **Google 0回率**を集計。

## 特別な経路

| 経路 | 挙動 |
|---|---|
| **心霊**（`isProprietaryOnly = effectiveDeepDive==="心霊"`） | Supabaseのみ・Google/Yahoo 0回。写真は利用者投稿(spot_photos)のみ。ダークUI＋免責バナー＋距離拡大ボタン |
| **お腹すいた** | 飲食店のみ。逆に飲食不要の気分では飲食店を出さない |
| **自由ワード / AI相談** | OpenAIがスポット提案 → Googleで実在確認(距離検証・閉店除外) → 5件未満なら構造化検索にフォールバック。近隣承認スポット14件をRAGで注入 |
| **レガシー経路** | Supabase-firstが0件等で失敗した時のGoogle/Yahoo主体のフォールバック。同じfinalizeで15件化 |

## コストの考え方

- **searchText（テキスト検索, ≈¥4〜5/回）が主コスト**。photo媒体取得(¥1)はほぼキャッシュで0。
- 削減レバー：①写真の `photo_url` 永続化（成熟するほど補完searchTextが0へ）②Supabase充足でGoogle/Yahooスキップ ③Yahoo取得数の抑制 ④マージでGoogle最小化。
- OpenAI判別/リランカーは gpt-4o-mini で **≈¥0.05/回** ＝ Google検索の約1/100。足してもほぼ無視できる。
